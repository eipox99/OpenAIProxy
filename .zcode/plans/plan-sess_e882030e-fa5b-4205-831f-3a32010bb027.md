# Model Sets — Implementation Plan

## Concept

Instead of client specifying a model name, they specify a **model set name** — a named, priority-ordered list of `(model_name, provider)` pairs. The proxy tries each entry in order, **rewriting the model name** when forwarding. If entry 1 fails, it falls through to entry 2, then 3, etc.

### Example

**Model set "freemodels":**
| Priority | Model forwarded | Provider |
|----------|----------------|----------|
| 1 | deepseek-v4-flash-free | OpenCode Zen |
| 2 | deepseek-v4-flash | DeepSeek |
| 3 | gpt-4o-mini | OpenAI |

Client sends: `{"model": "freemodels", "messages": [...]}`

Proxy tries:
1. Forward to OpenCode Zen with `model: "deepseek-v4-flash-free"` → if fails...
2. Forward to DeepSeek with `model: "deepseek-v4-flash"` → if fails...
3. Forward to OpenAI with `model: "gpt-4o-mini"` → if all fail, return error

**Model set "paidmodels":**
| Priority | Model forwarded | Provider |
|----------|----------------|----------|
| 1 | gpt-4o | OpenAI |
| 2 | claude-3-opus | Anthropic |

## Database — new tables

### `model_sets`
| Column | Type | Notes |
|--------|------|-------|
| id | INTEGER PK | |
| name | TEXT UNIQUE | e.g. "freemodels", "paidmodels" |
| is_default | BOOLEAN | Used if client sends unknown model name |
| created_at | DATETIME | |

### `model_set_entries`
| Column | Type | Notes |
|--------|------|-------|
| id | INTEGER PK | |
| model_set_id | INTEGER FK → model_sets.id (CASCADE) | |
| provider_id | INTEGER FK → providers.id (CASCADE) | |
| model_name | TEXT | The model to send in the forwarded request |
| priority | INTEGER | Order within the set |
| is_enabled | BOOLEAN | Toggle this entry without deleting |

Unique constraint: (model_set_id, priority)

### Keep existing tables (`providers`, `provider_models`, `usage_logs`)

`provider_models` is still useful for auto-detection — providers list their available models. But routing no longer uses `provider_models` for fallback; it uses `model_set_entries`.

## Routing Logic

In `provider_manager.py`, replace `get_available_providers(model_name)` with:

```
get_model_set_entries(set_name, session) → list of (model_name, provider) tuples

1. Query model_sets where name == set_name
2. If not found, use the default set (is_default == True)
3. Get all enabled entries sorted by priority
4. Filter out providers in cooldown
5. Return ordered list
```

In `proxy.py`, when forwarding a request:
1. Take the original request body
2. Create a copy with `model` rewritten to the entry's model_name
3. Forward to the entry's provider
4. On failure, try the next entry

## API Endpoints

### Model Set CRUD
| Method | Path | Description |
|--------|------|-------------|

| DELETE | /api/model-sets/{id} | Delete a set and its entries |

| Method | Path | Description |
|--------|------|-------------|

| PUT | /api/model-sets/{id}/entries/{entry_id}/toggle | Enable/disable entry |

## Default behavior

If the client sends `model: "gpt-4o"` (a specific model name, not a set name), the proxy:
1. Checks if "gpt-4o" is a model set name → if so, route through that set
2. If not, uses the **default** model set (the one with `is_default=True`)
3. But rewrites all entries' model names to "gpt-4o"... or just forwards the request body as-is?

Actually, simpler: **if the model field doesn't match a model set name, use the default set.** The entries in the default set each specify which model_name to forward. The client's requested model is used to match a set, not forwarded.

If the client sends `model: "gpt-4o"` and there's no set called "gpt-4o", use the default set. The entries in the default set will forward whatever model_name they specify.
